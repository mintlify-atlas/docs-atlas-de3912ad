---
title: "peer channel"
description: "Complete reference for the peer channel subcommands used to manage channel membership and configuration on a Hyperledger Fabric peer."
---

The `peer channel` command lets administrators manage channel-related operations on a peer: joining channels, fetching blocks, signing and submitting configuration updates, and querying channel information.

## Syntax

```bash
peer channel <subcommand> [flags]
```

Available subcommands:

| Subcommand | Description |
|---|---|
| `create` | Create a channel and write the genesis block to a file (deprecated) |
| `fetch` | Fetch a specific block from a channel |
| `getinfo` | Get blockchain metadata for a channel |
| `join` | Join the peer to a channel using a genesis block |
| `joinbysnapshot` | Join the peer to a channel from a snapshot |
| `joinbysnapshotstatus` | Query whether a snapshot join is in progress |
| `list` | List channels the peer has joined |
| `signconfigtx` | Sign a configuration transaction file |
| `update` | Sign and submit a configuration transaction to a channel |

## Global flags

All `peer channel` subcommands share these flags for orderer connectivity:

<ParamField path="--cafile" type="string">
  Path to a file containing PEM-encoded trusted certificate(s) for the ordering endpoint.
</ParamField>

<ParamField path="--certfile" type="string">
  Path to a file containing the PEM-encoded X509 public key for mutual TLS with the orderer.
</ParamField>

<ParamField path="--clientauth" type="boolean">
  Use mutual TLS when communicating with the orderer endpoint.
</ParamField>

<ParamField path="--connTimeout" type="duration" default="3s">
  Timeout for client to connect.
</ParamField>

<ParamField path="--keyfile" type="string">
  Path to a file containing the PEM-encoded private key for mutual TLS with the orderer.
</ParamField>

<ParamField path="-o, --orderer" type="string">
  Ordering service endpoint (`<hostname>:<port>`).
</ParamField>

<ParamField path="--ordererTLSHostnameOverride" type="string">
  Hostname override for TLS validation when connecting to the orderer.
</ParamField>

<ParamField path="--tls" type="boolean">
  Use TLS when communicating with the orderer endpoint.
</ParamField>

<ParamField path="--tlsHandshakeTimeShift" type="duration">
  Shift the time backwards for certificate expiration checks during TLS handshakes with the orderer.
</ParamField>

## peer channel create

<Warning>
  `peer channel create` is deprecated. Use the Orderer Service Node (OSN) admin commands to create channels instead.
</Warning>

Creates a channel and writes the genesis block to a file on disk.

**Syntax**

```bash
peer channel create [flags]
```

**Flags**

<ParamField path="-c, --channelID" type="string" required>
  The ID of the channel to create. Must be all lowercase, fewer than 250 characters, and match the pattern `[a-z][a-z0-9.-]*`.
</ParamField>

<ParamField path="-f, --file" type="string" required>
  Path to the configuration transaction file generated by `configtxgen`.
</ParamField>

<ParamField path="--outputBlock" type="string" default="./<channelID>.block">
  Path to write the genesis block for the newly created channel.
</ParamField>

<ParamField path="-t, --timeout" type="duration" default="10s">
  How long to wait for channel creation to complete.
</ParamField>

**Examples**

Create `mychannel` using a configuration transaction file:

```bash
peer channel create \
  -c mychannel \
  -f ./createchannel.tx \
  --orderer orderer.example.com:7050
```

Create a channel and wait up to 30 seconds:

```bash
peer channel create \
  -c mychannel \
  --orderer orderer.example.com:7050 \
  -f ./createchannel.tx \
  -t 30s
```

On success, block 0 (the genesis block) is returned and written to `mychannel.block` in the current directory.

## peer channel join

Joins the peer to a channel using the channel's genesis block.

**Syntax**

```bash
peer channel join [flags]
```

**Flags**

<ParamField path="-b, --blockpath" type="string" required>
  Path to the file containing the genesis block for the channel. Typically retrieved with `peer channel fetch`.
</ParamField>

**Example**

Join using the genesis block file retrieved via `peer channel fetch`:

```bash
peer channel join -b ./mychannel.genesis.block
```

```text
2018-02-25 12:25:26.511 UTC [channelCmd] InitCmdFactory -> INFO 003 Endorser and orderer connections initialized
2018-02-25 12:25:26.571 UTC [channelCmd] executeJoin -> INFO 006 Successfully submitted proposal to join channel
```

## peer channel fetch

Fetches a specified block from a channel and writes it to a file. Used to retrieve the genesis block before joining, or to inspect configuration blocks.

**Syntax**

```bash
peer channel fetch <newest|oldest|config|(number)> [outputfile] [flags]
```

The positional argument specifies which block to fetch:
- `newest` — the most recent block
- `oldest` — the first block (block 0)
- `config` — the most recent configuration block
- `<number>` — a specific block number

**Flags**

<ParamField path="-c, --channelID" type="string" required>
  The channel to fetch a block from.
</ParamField>

<ParamField path="--bestEffort" type="boolean">
  Ignore errors and return blocks on a best-effort basis.
</ParamField>

**Examples**

Fetch the newest block and save it to `mychannel.block`:

```bash
peer channel fetch newest mychannel.block \
  -c mychannel \
  --orderer orderer.example.com:7050
```

```text
2018-02-25 13:10:16.144 UTC [channelCmd] readBlock -> INFO 00a Received block: 32
```

Fetch block number 16:

```bash
peer channel fetch 16 \
  -c mychannel \
  --orderer orderer.example.com:7050
```

The block is saved as `mychannel_16.block` in the current directory.

## peer channel getinfo

Returns blockchain metadata for a channel, including the current block height and the cryptographic hashes of the most recent blocks.

**Syntax**

```bash
peer channel getinfo [flags]
```

**Flags**

<ParamField path="-c, --channelID" type="string" required>
  The channel to query.
</ParamField>

**Example**

```bash
peer channel getinfo -c mychannel
```

```text
2018-02-25 15:15:44.135 UTC [channelCmd] InitCmdFactory -> INFO 003 Endorser and orderer connections initialized
Blockchain info: {"height":5,"currentBlockHash":"JgK9lcaPUNmFb5Mp1qe1SVMsx3o/22Ct4+n5tejcXCw=","previousBlockHash":"f8lZXoAn3gF86zrFq7L1DzW2aKuabH9Ow6SIE5Y04a4="}
```

## peer channel signconfigtx

Signs a configuration transaction file in place. The file size increases as each organization's signature is appended. Collect all required signatures before calling `peer channel update`.

**Syntax**

```bash
peer channel signconfigtx [flags]
```

**Flags**

<ParamField path="-f, --file" type="string" required>
  Path to the configuration transaction file to sign. The file is modified in place.
</ParamField>

**Example**

Sign `updatechannel.tx` with the current peer's identity:

```bash
peer channel signconfigtx -f updatechannel.tx
```

The file size increases to reflect the appended signature:

```text
# Before
-rw-r--r--  1 user  staff   284 Feb 25 updatechannel.tx
# After
-rw-r--r--  1 user  staff  2180 Feb 25 updatechannel.tx
```

## peer channel update

Signs and submits a configuration transaction to the ordering service, updating the channel configuration.

**Syntax**

```bash
peer channel update [flags]
```

**Flags**

<ParamField path="-c, --channelID" type="string" required>
  The channel to update.
</ParamField>

<ParamField path="-f, --file" type="string" required>
  Path to the signed configuration transaction file to submit.
</ParamField>

**Example**

Update `mychannel` with a signed configuration transaction:

```bash
peer channel update \
  -c mychannel \
  -f ./updatechannel.tx \
  -o orderer.example.com:7050
```

```text
2018-02-23 06:32:11.569 UTC [channelCmd] InitCmdFactory -> INFO 003 Endorser and orderer connections initialized
2018-02-23 06:32:11.626 UTC [main] main -> INFO 010 Exiting.....
```

<Note>
  All organizations required by the channel's mod policy must have signed the configuration transaction before submitting it with `peer channel update`. Use `peer channel signconfigtx` on each required admin peer before submitting.
</Note>

## peer channel joinbysnapshot

Joins the peer to a channel using a snapshot created by another peer. The snapshot must have already been generated and transferred to the local filesystem.

**Syntax**

```bash
peer channel joinbysnapshot [flags]
```

**Flags**

<ParamField path="--snapshotpath" type="string" required>
  Path to the local snapshot directory.
</ParamField>

**Example**

Join using a snapshot in `/snapshots/completed/testchannel/1000`:

```bash
peer channel joinbysnapshot \
  --snapshotpath /snapshots/completed/testchannel/1000
```

```text
2020-10-12 11:41:45.442 EDT [channelCmd] executeJoin -> INFO 002 Successfully submitted proposal to join channel
2020-10-12 11:41:45.444 EDT [channelCmd] joinBySnapshot -> INFO 003 The joinbysnapshot operation is in progress. Use "peer channel joinbysnapshotstatus" to check the status.
```

<Warning>
  While a `joinbysnapshot` operation is in progress, no other `peer channel join` or `peer channel joinbysnapshot` commands can be run simultaneously on the same peer.
</Warning>

## peer channel joinbysnapshotstatus

Queries whether a `joinbysnapshot` operation is currently in progress on the peer.

**Syntax**

```bash
peer channel joinbysnapshotstatus
```

This subcommand takes no subcommand-specific flags.

**Examples**

When a join is in progress:

```bash
peer channel joinbysnapshotstatus
```

```text
A joinbysnapshot operation is in progress for snapshot at /snapshots/completed/testchannel/1000
```

When no join is in progress:

```text
No joinbysnapshot operation is in progress
```
