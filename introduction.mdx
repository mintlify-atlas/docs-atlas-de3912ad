Hyperledger Fabric is an open-source, enterprise-grade permissioned distributed ledger technology (DLT) platform hosted under the Linux Foundation. Unlike public blockchains such as Bitcoin or Ethereum, Fabric is designed from the ground up for enterprise use cases where participant identity, data confidentiality, and high transaction throughput are critical requirements.

## What is Hyperledger Fabric?

Fabric provides a uniquely **modular and configurable** architecture that enables innovation, versatility, and optimization for a broad range of industry use cases — from banking and finance to supply chain, healthcare, and digital assets.

Key differentiators from other blockchain platforms:

- **Permissioned network** — All participants are known and authenticated using X.509 certificates issued by a Certificate Authority (CA). There are no anonymous actors.
- **Execute-order-validate** — Fabric's unique transaction flow separates execution from ordering, enabling parallel processing, standard programming languages, and high throughput.
- **No cryptocurrency required** — Fabric does not need a native token or mining to operate, removing significant cost and attack vectors.
- **Pluggable components** — Consensus (Raft CFT or SmartBFT), identity management, and cryptographic libraries are all swappable without re-architecting the network.
- **Privacy by design** — Channels and private data collections allow subsets of network participants to transact confidentially, even on a shared infrastructure.

## Core components

| Component | Role |
|-----------|------|
| **Peer** | Hosts the ledger and executes chaincode (smart contracts) |
| **Orderer** | Orders transactions into blocks and delivers them to peers |
| **Certificate Authority (CA)** | Issues X.509 certificates for identity management |
| **Chaincode** | Business logic running inside isolated containers |
| **Channel** | A private sub-network for transaction isolation |
| **MSP** | Membership Service Provider — maps certificates to identities |

## The execute-order-validate model

Fabric's transaction flow has three distinct phases:

1. **Execute** — Client sends a transaction proposal to endorsing peers, which simulate execution and return a signed read-write set
2. **Order** — The signed transaction is sent to the ordering service, which produces a deterministic sequence of blocks
3. **Validate** — All peers validate endorsement policies and commit valid transactions to the ledger

This approach eliminates non-determinism (allowing standard languages like Go, Java, and Node.js), enables parallel execution across peers, and preserves confidentiality since only endorsing peers see the transaction data.
